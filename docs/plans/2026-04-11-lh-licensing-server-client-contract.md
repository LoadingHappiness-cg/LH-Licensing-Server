# LH.Licensing.Server Public Client Contract

Version: `1.0`
Audience: desktop apps and future clients of the Loading Happiness licensing platform

## Scope

This document defines the public contract consumed by client applications:

- `POST /api/licenses/activate`
- `POST /api/licenses/refresh`
- JWT claims emitted by the server
- local validation rules in the client
- expected error codes and status codes
- offline and rotation behavior

The contract is intentionally product-agnostic. Specific product behavior must be expressed through `productCode`, `appId`, `planCode`, and `entitlements`, not through app-specific branches in the server.

## Common conventions

- All timestamps are UTC and must be serialized in ISO 8601 format with `Z`, for example `2026-04-11T12:34:56.789Z`.
- JWT time claims (`iat`, `nbf`, `exp`, `offline_grace_until`) are numeric Unix epoch seconds.
- The client must treat tokens as opaque strings and must not attempt to parse refresh token contents.
- The refresh token is rotated on every successful refresh. The previous refresh token becomes invalid immediately after the server responds.

## `POST /api/licenses/refresh`

### Request

```json
{
  "refreshToken": "3r3bQx8xkQ4f7G1vP7r2dQ",
  "productCode": "LH.DESKTOP.SAMPLE",
  "appId": "lh.labels.gs1.desktop",
  "machineFingerprint": "fingerprint-001",
  "clientVersion": "1.0.1"
}
```

### Response

```json
{
  "accessToken": "eyJhbGciOiJSUzI1NiIsImtpZCI6InRlc3Qta2V5LTEifQ....",
  "refreshToken": "GvY0nJ9Tg8Xlqv7c1qkZqA",
  "expiresAt": "2026-04-11T13:34:56.789Z",
  "offlineGraceUntil": "2026-04-18T12:34:56.789Z",
  "policyVersion": 1,
  "entitlements": {
    "maxActivations": 2,
    "offlineGraceDays": 7,
    "features": ["basic", "standard"]
  }
}
```

### Refresh rotation rules

- If refresh succeeds, the new refresh token replaces the old one immediately.
- The old refresh token must be discarded by the client.
- A refresh token can only be used once.
- If the client retries with the old token after a successful refresh, the server may reject it as invalid or not found.

## JWT claims

### Standard claims

- `iss`: issuer URL of the licensing server.
- `aud`: audience string identifying the licensing API.
- `sub`: activation identifier, not the user identity.
- `jti`: unique JWT identifier for the issued token.
- `iat`: issued-at time.
- `nbf`: not-before time.
- `exp`: expiration time.

### Licensing claims

- `license_id`: license identifier.
- `customer_id`: customer identifier owning the license.
- `product_id`: product identifier.
- `product_code`: stable product code.
- `app_id`: stable client app identifier.
- `installation_id`: installation identifier.
- `plan_code`: plan code applied to the license.
- `policy_version`: version of the license policy snapshot used to issue the token.
- `offline_grace_until`: latest UTC timestamp allowed for offline use without refresh.
- `license_status`: current license status at token issuance time.
- `entitlements`: JSON object containing the client-facing feature and quota snapshot.

### Meaning of the licensing claims

- `product_code` and `app_id` let the client verify it received a token for the correct app/product pairing.
- `installation_id` binds the token to the machine fingerprint used during activation.
- `plan_code`, `policy_version`, and `entitlements` describe the runtime license capabilities.
- `offline_grace_until` defines the latest acceptable offline window.
- `license_status` allows the client to detect revoked or suspended states if the token is still locally cached.

## Local client validation

The client must validate locally:

1. JWT signature using the configured public key.
2. `iss` matches the configured issuer.
3. `aud` matches the configured audience.
4. `exp` is in the future.
5. `nbf` is not in the future beyond the configured clock skew.
6. `product_code` matches the current app configuration.
7. `app_id` matches the current app identity.
8. `installation_id` matches the local installation state.
9. `license_status` is still usable.
10. `offline_grace_until` is not exceeded for offline mode.

The client must block or degrade if any of the following is true:

- signature validation fails;
- `exp` is expired and the client cannot refresh;
- `offline_grace_until` is exceeded;
- `license_status` is `Revoked`, `Suspended`, or otherwise unusable;
- `product_code` or `app_id` does not match the current app;
- the local installation fingerprint changes unexpectedly.

Recommended behavior:

- `exp` expired but still within offline grace: allow a controlled offline session and attempt refresh as soon as connectivity returns.
- `offline_grace_until` exceeded: block core features.
- revoked license: block immediately on next online check or at the next local validation boundary.

## Error contract

All functional failures return a `ProblemDetails` payload with a stable `errorCode`.

### Activation and refresh errors

| HTTP status | `errorCode` | Meaning |
|---|---|---|
| `400` | `product_not_found` | Product code is unknown |
| `400` | `app_not_allowed` | App is not allowed for that product |
| `400` | `license_not_found` | License key does not exist |
| `400` | `license_hash_mismatch` | License key hash check failed |
| `400` | `license_inactive` | License is inactive or revoked |
| `400` | `license_out_of_range` | License is outside validity dates |
| `400` | `plan_invalid` | License plan is inactive or invalid |
| `400` | `activation_limit_reached` | Maximum activations exceeded |
| `400` | `refresh_token_not_found` | Refresh token is unknown or was rotated away |
| `400` | `refresh_token_expired` | Refresh token is no longer valid |
| `400` | `activation_revoked` | Activation or license has been revoked |
| `400` | `fingerprint_mismatch` | Fingerprint does not match the activation |
| `400` | `app_mismatch` | App ID does not match the activation |
| `400` | `product_mismatch` | Product does not match the activation |
| `400` | `activation_invalid` | Activation is incomplete or malformed |

### Admin errors

| HTTP status | `errorCode` | Meaning |
|---|---|---|
| `404` | `license_not_found` | License ID does not exist |

## Time and offline guidance

- The server emits UTC times only.
- The client should store the last successful JWT and refresh token securely.
- The client should retry refresh before `exp` if online.
- If the client is offline, it should continue until `offlineGraceUntil`.
- Once the grace window expires, the client should block or degrade according to product policy.

## Example operating model

- Activate once when the user enters a valid key.
- Refresh periodically when the app starts or before `exp`.
- Replace the refresh token on every successful refresh.
- If the server revokes the license, the next refresh fails and the client must stop treating the license as valid.
